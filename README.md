# Групповой проект: TaskFlow

## Описание проекта

TaskFlow — это программное обеспечение для управления задачами, предназначенное для командной работы. Приложение решает прикладную проблему эффективного создания типовых задач, удобной навигации по общему списку задач и прозрачного отслеживания статуса выполнения. Программа использует архитектуру MVC, что позволяет разделить логику, данные и интерфейс, а также применяет три паттерна проектирования: Prototype (для клонирования шаблонов задач), Iterator (для последовательного перехода между задачами) и State (для управления состояниями задач). Хранение данных реализовано в структурированном текстовом файле (tasks.txt с разделителем "|").

## Описание задания

Разработать программное обеспечение для управления задачами с тремя функциями - создание задач, совместная работа команды и отслеживание выполнения. Программа должна использовать архитектуру MVC и применять паттерн Prototype для клонирования шаблонов задач, паттерн Iterator для перехода от одной задачи к другой и паттерн State для управления различными состояниями задач. В качестве базы данных для хранения данных о задачах должен использоваться структурированный текстовый файл.

## Описание работы с GIT

Ветка main изначально содержит Readme.md с описанием постановки задачи и файл .gitignore для сохранения только полезных данных. Все мерджи обязательно проходят ревью через пулл-реквест. Правила нейминга выбраны следующим образом:

1. main - ветка для хранения стабильного кода. Мердж Тим-лидом без пулл-реквеста.
2. test - ветки для хранения стабильного кода бизнес-задачи. Мердж через пулл-реквест и код-ревью.
3. dev - ветка для хранения нестабильного кода бизнес-задачи и любых промежуточных процессов разработки
4. test_unit - ветка для написания и добавления unit-тестов
5. если реализацией одной бизнес-задачи занимается несколько человек, то в именах веток ниже test допускается явное указание имя разработчика, например ivanov.ivan_dev_module-name
6. В случае нарушения нейминга пулл-реквест отклоняется без ревью кода.

## Бизнес-задача №1. Создание задач с использованием Prototype

Пользователи должны иметь возможность создавать новые задачи на основе существующих шаблонов. Пользователь может создать шаблон задачи (название, описание, дата выполнения, приоритет) и клонировать его для быстрого создания новых задач без повторного ввода данных. Порождающий паттерн для применения - это паттерн Prototype, который будет использоваться для клонирования шаблонов задач. Этот паттерн позволит пользователям создавать типовые задачи и массово генерировать похожие задания.

Ответственный: Устинов Егор. Ветвь бизнес-задачи: test_task-creation

## Бизнес-задача №2. Совместная работа команды с использованием Iterator

Пользователи должны иметь возможность последовательно переходить от одной задачи к другой в общем списке. Команда может просматривать задачи по очереди, передавать их друг другу и перемещаться по списку с помощью навигации "следующая задача" / "предыдущая задача". Поведенческий паттерн для применения - это паттерн Iterator, который будет использоваться для перебора коллекции задач без раскрытия её внутренней структуры. Этот паттерн позволит пользователям удобно перемещаться между задачами и работать с ними в заданном порядке.

Ответственный: Щербакова Кристина. Ветвь бизнес-задачи: test_collaboration

## Бизнес-задача №3. Отслеживание выполнения с использованием State

Пользователи должны иметь возможность отслеживать ход выполнения каждой задачи и изменять её состояние. Каждая задача может находиться в одном из состояний: новая, в работе, на проверке, завершена. Поведенческий паттерн для применения - это паттерн State, который будет использоваться для управления различными состояниями задач. Этот паттерн позволит пользователям изменять статус задачи и автоматически ограничивать доступные действия в зависимости от текущего состояния.

Ответственный: Бакланова Ляна. Ветвь бизнес-задачи: test_progress-tracking

## Unit-тесты

Все методы классов покрыты unit-тестами. Тесты написаны с использованием фреймворка xUnit. Ветка для разработки тестов: test_unit. После завершения написания тестов выполняется пулл-реквест в ветку test.

### Результат работы

Программа должна представлять собой консольное или оконное приложение, позволяющее пользователю:
- создавать шаблоны задач и клонировать их
- просматривать список задач с возможностью последовательного перехода между ними
- изменять состояние каждой задачи
- сохранять и загружать данные из текстового файла
